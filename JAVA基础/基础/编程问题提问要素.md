 🎯 超级提问框架：CLEAR 模型

这是一个**适用于任何技术/学习问题**的提问 SOP，能大幅提升回答质量。

## 核心原则

> **AI 的回答质量 = 问题的结构化程度 × 背景信息完整度**

越结构化、越具体的问题 → 越能得到可执行的答案

---

## 📋 CLEAR 五步提问法

| 步骤 | 含义 | 核心动作 | 检查点 |
|:---|:---|:---|:---|
| **C - Context** | 上下文 | 说明你的背景、目标、已知内容 | ✅ 回答者能判断你的水平吗？<br>✅ 说明了你想达成什么吗？ |
| **L - Level** | 定位层次 | 精准说明卡在哪个环节（理解/实现/优化） | ✅ 是概念不懂，还是实现不了？<br>✅ 是完全不会，还是部分困难？ |
| **E - Expectation** | 期望形式 | 明确想要什么类型的答案 | ✅ 要方法论/例子/检查清单？<br>✅ 要深度分析还是快速指南？ |
| **A - Attempt** | 已尝试的 | 说明你做过什么、为什么没成功 | ✅ 展示了思考过程吗？<br>✅ 避免得到重复建议 |
| **R - Refinement** | 边界细化 | 提供边界案例、反例、约束条件 | ✅ 说明了适用场景吗？<br>✅ 有具体例子吗？ |

---

## 🔥 可直接套用的提问模板

### 模板 1: 学习概念类

```
【背景】我目前在学 [主题]，已经了解 [已知内容]
【困难】但在 [具体环节] 遇到困难：[精准描述问题]
【尝试】我尝试过 [方法A/方法B]，但 [为什么失败]
【期望】希望得到 [方法论/步骤清单/可复用框架]，而不只是 [不想要的答案形式]
【场景】举个例子：[具体案例说明困难点]
```

**示例**：
```
【背景】我在学习 Java 面向对象，已经理解类和对象的基本概念
【困难】但在设计多个类之间的关系时不知道如何下手：该用继承、组合还是接口？
【尝试】我看了继承的例子，但一写代码就变成"为了继承而继承"，感觉很僵硬
【期望】希望有一个可复用的决策流程（类似流程图），帮我判断什么时候该用什么，而不只是"看情况而定"这种建议
【场景】比如设计一个图书管理系统，有 Book、EBook、User、Admin，我不知道它们之间该如何组织
```

### 模板 2: 调试/实现类

```
【目标】我想实现 [功能]
【现状】目前代码 [能做到什么/不能做到什么]
【困难】具体卡在 [错误现象/不理解的点]
【尝试】我试过 [方法]，结果是 [现象]，我猜测原因是 [你的理解]
【期望】想知道 [正确的思路/调试方法/底层原理]
【约束】[如果有] 环境限制、不能使用的方案
```

### 模板 3: 方法论/最佳实践类

```
【场景】我经常需要 [做某件事]
【问题】目前的做法是 [当前方法]，但存在 [具体问题]
【期望】有没有 [系统的方法论/检查清单/标准流程]，可以让我 [达到什么效果]
【对比】我知道有 [方法A] 和 [方法B]，但不知道如何选择
【应用】我想用在 [具体场景]，而不只是理论学习
```

## ⚡ 提问质量速查清单

在点击发送前，检查这 7 个要素：

- [ ] **1. 说明了你的水平** - 新手/中级/已工作？学到哪个章节了？
- [ ] **2. 精准定位卡点** - 不是"学不会"，而是"理解了X，但Y环节不会"
- [ ] **3. 展示思考过程** - 说明你试过什么、为什么认为不对
- [ ] **4. 明确期望形式** - 要步骤/要原理/要例子/要检查清单？
- [ ] **5. 提供具体案例** - 有边界案例、反例、或真实代码片段
- [ ] **6. 说明应用场景** - 做作业/准备面试/解决工作问题？
- [ ] **7. 避免XY问题** - 你问的是真正的问题，还是你认为的解决方案？

## 📊 提问质量对比（实战示例）

### 场景：学习数据库索引

#### ❌ 低质量提问
```
"索引怎么用？"
```
**问题**：
- 没有背景，不知道你的水平
- 范围太大，可能得到教科书式的回答
- 不知道你具体卡在哪

---

#### ⭐ 中等提问（你之前的水平）
```
"我知道索引能加快查询，但不知道什么时候该建索引，什么时候不该建"
```
**优点**：
- ✅ 定位了具体问题（判断时机）
- ✅ 说明了已知内容

**可改进**：
- 缺少期望形式
- 没有具体场景

---

#### 🚀 超级提问
```
【背景】我在学数据库优化，已经理解索引是通过 B+ 树加速查询的原理
【困难】但在实际项目中，不知道如何系统性地判断哪些字段该建索引
【尝试】我看到建议"给查询条件建索引"，但我的表有 10 个字段，WHERE 条件可能用任意组合，难道都建索引吗？我试过给所有字段建，结果插入变慢了
【期望】希望有一个决策流程或检查清单，帮我判断：
  - 哪些场景必须建索引？
  - 哪些场景不该建？
  - 如何权衡查询速度和写入速度？
  而不只是"看具体情况"这种答案
【场景】比如一个用户表（100万行），有 id、username、email、age、city、created_at 等字段，常见查询是"按 city 和 age 范围查询"，我该如何设计索引策略？
```

**为什么是超级提问**：
- ✅ 背景完整（学到哪了、理解了什么）
- ✅ 问题精准（不是"怎么用"，是"如何决策"）
- ✅ 展示思考（尝试过、发现了问题）
- ✅ 期望明确（要决策流程，不要模糊建议）
- ✅ 有具体案例（100万行、具体字段、具体查询）

**会得到的回答**：
- 可能是一个决策树（流程图）
- 可能是三层检查清单（必建/可选/禁建）
- 可能是针对你的案例的具体索引设计 + 通用方法论

## 🎓 进阶技巧：元提问

当你不确定如何问一个复杂问题时，可以**先问如何问**：

```
"我想学习 [主题]，但这个话题很复杂，我不确定该如何有效提问。我目前的水平是 [背景]，我的困难是 [大致方向]。你能帮我把问题结构化吗？或者告诉我应该关注哪些关键点？"
```

**示例**：
```
"我想深入理解 JVM 内存管理，但这个话题太大了。我目前知道堆、栈的基本概念，但对 GC 算法、内存溢出定位等还很模糊。你能帮我梳理一下，我应该按什么顺序学习？每个环节应该问哪些关键问题？"
```

## 🔑 核心心法

### 1. **具体化金字塔**
```
模糊问题 → 具体领域 → 精准环节 → 边界案例
   ↓           ↓           ↓          ↓
 "循环"    "嵌套循环"   "找边界关系"  "三角形第3行"
```

### 2. **期望形式清单**
明确告诉 AI 你想要的答案类型：
- 📝 **方法论/框架**："给我可复用的步骤"
- ✅ **检查清单**："给我自查列表"
- 📊 **对比分析**："A 和 B 的适用场景"
- 🎯 **决策树**："如何选择最佳方案"
- 💡 **原理解释**："为什么会这样"
- 🔧 **实操指南**："具体怎么做"

### 3. **避免 XY 问题**
不要问"如何用方法 Y"，而要问"如何解决问题 X"

❌ "Java 中如何用单例模式？"（这是你认为的方案）
✅ "我需要全局唯一的配置管理器，但不知道如何设计，避免被重复创建"（这是真正的问题）

## 📌 快速启动：三个万能句式

记住这三句话，立刻提升 80% 的提问质量：

1. **"我已经理解了 X，但在 Y 环节不知道 Z"**
    - 说明边界、定位卡点

2. **"希望得到 [形式]，而不只是 [不想要的]"**
    - 引导答案类型

3. **"举个例子：[具体案例]，我的困难是 [现象]"**
    - 提供锚点、避免抽象

---

## 🎯 总结

高质量提问 = **CLEAR 模型** + **速查清单** + **明确期望**

记住：
- ✅ **结构化**优于流水账
- ✅ **具体案例**优于抽象描述
- ✅ **展示思考**优于直接求答案
- ✅ **明确期望**优于开放式问题

下次提问前，用这个清单检查一遍，你会发现回答质量显著提升！
