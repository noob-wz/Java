 **「嵌套类」（Nested Class）**，也就是把一个类定义在另一个类的内部。

在 `ProductDemo` 中定义 `static class Item`，主要是出于以下两个原因：

### 1. 为了教学和演示的便利（主要原因）

如果我不把 `Item` 放在里面，而是写成标准的独立类，那么这一个示例就需要拆成两个文件：

* `Item.java`
* `ProductDemo.java`

这就导致你无法直接复制一段代码就运行。
**把它们合在一起，你可以直接复制这一整块代码到 IDE 中，马上就能跑起来，降低了上手的麻烦。**

### 2. 逻辑上的从属关系

有时候，某个类（比如 `Item`）只为另一个类（`ProductDemo`）服务，其他地方根本用不到它。这时候把它"藏"在内部，可以保持外部环境的整洁。

我们可以用**文件文件夹**来类比：

```text
[标准的写法 - 两个文件]
📁 项目文件夹
    ├─ 📄 ProductDemo.java  (只负责运行 main)
    └─ 📄 Item.java         (只负责定义商品)

---------------------------------------

[现在的写法 - 一个文件]
📄 ProductDemo.java
    |
    |-- class ProductDemo
    |     |
    |     |-- static class Item (就像把 Item 文件塞进来了)
    |     |
    |     |-- main()

```

---

### 💡 核心知识点：为什么要加 `static`？

注意看，我写的是 `static class Item`。

* **如果不加 static**：`Item` 就变成了"内部类"，它必须依赖于 `ProductDemo` 的**实例对象**才能创建。也就是说，你得先 new 一个 `ProductDemo`，才能 new 一个 `Item`。
* **加了 static**：它变成了"静态嵌套类"。虽然它写在里面，但在地位上它和外面的类几乎是一样的。**可以在 `main` 方法（也是 static 的）里直接 `new Item(...)`，完全不需要关心 `ProductDemo` 的对象是否存在。**

### ✅ 最佳实践建议

* **在练习/Demo时**：为了代码紧凑，经常使用静态内部类。
* **在真实项目中**：除非 `Item` 真的只有 `ProductDemo` 会用到，否则**通常建议把 `Item` 提取出来，单独存为一个 `Item.java` 文件**。这符合"单一职责原则"，也方便其他类使用它。

---

**一句话总结**：
这里这样写是为了让你**"一次复制，直接运行"**，在真实的工程开发中，我们通常会把它们分成两个文件写。
