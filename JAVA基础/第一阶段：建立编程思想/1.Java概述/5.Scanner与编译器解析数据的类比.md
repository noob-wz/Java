## 程序文件存储、文本编辑和数据解析之间的本质区别

首先，**编辑器在保存程序文件时，存储的是字符（'1', '2', '3' 三个字符），而不是数值 123。**

计算机并不知道这是一个“数”，除非有特定的程序（如编译器或解释器）来处理它。

---

### 1. 📝 编辑器与程序文件 (字符存储)

当在 VS Code、Notepad++ 或任何文本编辑器中输入 `int x = 123;` 并保存时：

- **存储内容：** 文件中存储的是文本流。数字 `123` 占据了三个字节（或更多，取决于编码，但通常是 3 个字节），分别对应字符 `'1'`、字符 `'2'`、字符 `'3'` 的编码值（例如 ASCII/UTF-8 码：49, 50, 51）。
- **计算机的视角：** 在这一阶段，计算机只知道这三个字节是按照某种编码规则排列的，它将这三个字节视为**文本数据**。
- **用途：** 这种存储方式是为了让人类（程序员）能够阅读和编辑源代码。

### 2. 🤖 编译与解析 (数值处理)

计算机真正知道 `123` 是一个数字，并能够进行数学运算，是在**编译或运行时**。

这里有一个专门的程序来做这个处理，它比 `Scanner` 更复杂、更核心：**编译器 (Compiler) 或运行时解释器 (Runtime Interpreter)**。

| 角色              | 职责                   | 处理过程                                                                                                                                                                                             |
| :---------------- | :--------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **源代码**        | `int x = 123;`         | 存储在硬盘上的是字符流 ('i', 'n', 't', ' ', 'x', ' ', '=', ' ', '1', '2', '3', ';')                                                                                                                  |
| **编译器/解释器** | **语法分析与词法分析** | 1. 发现字符序列 '1', '2', '3'。 2. 识别出这个序列是一个**数字字面量 (Numeric Literal)**。 3. 将这三个字符**转换**成 32 位的整数值 `0000 0000 0000 0000 0000 0000 0111 1011`（即数值 123 的二进制）。 |
| **运行内存**      | **存储数据**           | 当程序运行时，变量 `x` 在内存中占据 4 个字节，直接存储的是数值 123 的**二进制补码**形式，而不是三个字符的编码。                                                                                      |

### 💡 `Scanner` 与编译器的类比

**有处理过程，而且这个处理过程和 `Scanner` 的原理非常相似：**

- **相似点 (解析目的)：** `Scanner` 负责将用户输入（也是字符流）转换为基本类型；编译器负责将源代码中的数字字面量（也是字符流）转换为基本类型。
- **不同点 (执行阶段)：**
  - `Scanner` 在程序 **运行时 (Runtime)** 执行，处理来自键盘或文件的**输入数据**。
  - 编译器在程序 **编译时 (Compile Time)** 执行，处理**源代码**。

简而言之：**编辑器只管记录你按了什么键（字符），而编译器/解释器负责像 `Scanner` 一样，将这些字符翻译成计算机能理解和运算的数值。**
