# Java 包命名规范核心笔记

## 1. 核心铁律 (Mandatory Rules)

### 1.1 全部小写 (All Lowercase)

* **规则：** 包名中的字母**必须全部小写**。
* **原因：**
  1. **避免冲突：** 为了和类名（大驼峰 `PascalCase`）及变量名（小驼峰 `camelCase`）区分开。
  2. **跨平台兼容：** Windows 文件名不区分大小写，而 Linux/Unix 区分。如果包名大小写混用，在不同系统间迁移代码时（如从 Windows 开发环境部署到 Linux 服务器）会引发致命的 `ClassNotFoundException`。


* **示例：**
* ✅ `com.example.project`
* ❌ `com.Example.Project`



### 1.2 点分隔符 (Dot Separator)

* **规则：** 使用点 `.` 来划分目录层级。每个点代表一层文件夹。
* **示例：** `java.util` 对应硬盘路径 `java/util/`。

---

## 2. 结构标准：倒置域名法 (Reverse Domain Name)

这是全球通用的标准，保证了包名的**全球唯一性**。

**通用格式：**
`前缀.公司/组织名.项目名.模块名.分层名`

### 2.1 第一级：顶级域名 (Top-Level Domain)

* `com` : 商业机构 (Company)，如 `com.alibaba`。
* `org` : 非营利组织 (Organization) 或开源项目，如 `org.apache`。
* `gov` : 政府机构 (Government)。
* `edu` : 教育机构 (Education)。
* `net` : 网络服务商。

### 2.2 第二级：公司/作者名

* 也就是域名的主体。
* **示例：** `com.google`, `com.xiaomi`, `me.lixiaolai` (个人项目常用 `me` 或 `io`)。

### 2.3 第三级：项目名 (Project Name)

* 当前开发的项目名称。
* **示例：** `com.xiaomi.mall` (小米商城项目)。

### 2.4 第四级：功能模块 (Module) - *可选*

* 如果项目很大，会按业务拆分。
* **示例：** `com.xiaomi.mall.user` (用户模块), `com.xiaomi.mall.order` (订单模块)。

### 2.5 第五级：架构分层 (Layer)

* 代码的具体功能分类。
* **示例：** `com.xiaomi.mall.user.dao` (用户模块的数据库访问层)。

---

## 3. 常用分层包名“词典” (Standard Vocabulary)

在项目末尾，通常会根据类的功能进行归类。以下是行业内约定俗成的命名：

| 包名后缀 | 含义 (Meaning) | 存放内容 |
| --- | --- | --- |
| **`.web`** / **`.controller`** | 前端控制器 | 处理 HTTP 请求，负责与前端交互的入口。 |
| **`.service`** | 业务逻辑层 | 核心业务代码（如：转账逻辑、下单逻辑）。 |
| **`.dao`** / **`.mapper`** | 数据访问层 | (Data Access Object) 专门负责操作数据库（增删改查）。 |
| **`.entity`** / **`.model`** / **`.bean`** | 实体类 | 对应数据库表的类（如 `User`, `Product`）。 |
| **`.util`** / **`.common`** | 工具类 | 通用的、非业务的工具（如 `DateUtil`, `StringUtil`）。 |
| **`.config`** | 配置类 | 存放系统配置（如 `AppConfig`, `DatabaseConfig`）。 |
| **`.dto`** | 数据传输对象 | (Data Transfer Object) 前端传给后端的参数对象。 |
| **`.vo`** | 视图对象 | (View Object) 后端传给前端用于展示的对象。 |
| **`.impl`** | 接口实现 | 存放接口的具体实现类（通常藏在 service 或 dao 之下）。 |

---

## 4. 避坑指南 (Anti-Patterns)

### 4.1 严禁使用连字符 (-)

* **错误：** `com.my-project.utils`
* **原因：** Java 语法不支持减号作为标识符。
* **修正：**
* 直接拼接：`com.myproject.utils` (推荐)
* 下划线：`com.my_project.utils` (极少用，不推荐)



### 4.2 避开 Java 关键字

* **场景：** 假如你的域名叫 `int.com`，或者模块叫 `class`。
* **错误：** `com.int.utils` (`int` 是关键字，编译器会报错)。
* **修正：** 在关键字后加下划线。
* ✅ `com.int_.utils`



### 4.3 拒绝以数字开头

* **错误：** `com.10086.mobile`
* **原因：** Java 标识符不能以数字开头。
* **修正：** 加前缀，如 `com.chinamobile10086`。

### 4.4 单复数问题 (Singular vs Plural)

* **争议：** 是用 `util` 还是 `utils`？是 `model` 还是 `models`？
* **建议：** 两种都可以，**但必须在一个项目中保持统一**。
* *趋势：* 目前主流开发规范（如阿里规范）倾向于使用**单数**（`util`, `service`, `dao`），但 `utils` 作为工具包非常常见，作为例外已被接受。

* **物理路径：** `.../src/com/google/maps/search/util/GeoUtil.java`
* **代码声明：**

```java
package com.google.maps.search.util; // 👈 完美符合规范

public class GeoUtil {
    // ...
}

```
